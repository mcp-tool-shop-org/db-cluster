## 4. CRITICAL findings (8)

### KERNEL-001 — `PolicyEnforcedKernel` is missing wrappers for 7 ClusterKernel verbs
**File:** `src/kernel/policy-enforced-kernel.ts`
**Description:** Seven public methods on `ClusterKernel` have **no** wrapper in `PolicyEnforcedKernel`: `ingestArtifact`, `createEntity`, `linkEvidence`, `traceProvenance`, `indexStatus`, `traceBundle`, `explainTrace`. Any caller that obtains a `ClusterKernel` reference (e.g. the `@internal _kernel` getter at `policy-enforced-kernel.ts:347`, or directly via `src/cli.ts` which calls all of these as `kernel.ingestArtifact(...)`, `kernel.createEntity(...)`, etc.) can perform full writes to canonical/artifact stores and emit provenance with **zero** policy check and **zero** redaction. `ingestArtifact` + `createEntity` + `linkEvidence` are primary write paths.
**Recommendation:** Either (a) add wrappers in `PolicyEnforcedKernel` for all 7 missing verbs (each calling `enforce()` with the right capability), or (b) remove the 3 write helpers from `ClusterKernel` entirely and force all writes through `proposeMutation → commitMutation` (also fixes KERNEL-002). Option (b) is more consistent with "every mutation crosses a typed command boundary." Either way, document that the `@internal _kernel` getter is for tests only and audit production call sites.
**Evidence:** Grep `async\s+(ingestArtifact|createEntity|linkEvidence|traceProvenance|indexStatus|traceBundle|explainTrace)` on `policy-enforced-kernel.ts` returns 0 matches; same grep on `cluster-kernel.ts` returns matches for all 7. `cli.ts:64` `kernel.ingestArtifact(...)`, `cli.ts:92` `kernel.createEntity(...)`, `cli.ts:114` `kernel.linkEvidence(...)`, `cli.ts:356` `kernel.indexStatus()`, `cli.ts:506/534/554` `kernel.explainTrace(...)`, `cli.ts:547` `kernel.traceBundle(...)`.

### KERNEL-003 — `PolicyEnforcedKernel.findSources` returns `indexRecords` unfiltered
**File:** `src/kernel/policy-enforced-kernel.ts:189`
**Description:** `findSources` filters `resolvedEntities` and `resolvedArtifacts` by per-object policy check (lines 151–186), but returns `indexRecords: result.indexRecords` UNFILTERED at line 189. The `IndexRecord` contains `sourceId`, `text` (constructed as `${kind}: ${name}` for entities), and `metadata` (containing all entity attributes for entities — see `cluster-kernel.ts:178`). When a principal is denied `read_owner_truth` on a restricted entity, they nevertheless see the entity's URI, kind, name, and all attributes via the indexRecord. This defeats the per-entity filtering immediately above and leaks restricted truth via the discovery surface.
**Recommendation:** Filter `indexRecords` using the same per-object policy + visibility logic. For each `indexRecord`, build the candidate URI (`cluster://{sourceStore}/{sourceId}`), check `read_derivative` + visibility, and exclude denied records. Consider whether `sourceStore='canonical'` index records should be hidden whenever the corresponding `read_owner_truth` is denied (since the index text mirrors the canonical state).
**Evidence:** `policy-enforced-kernel.ts:188-192` — return statement explicitly passes through `result.indexRecords` with no filter. `cluster-kernel.ts:174-179` (createEntity) confirms that index text and metadata mirror entity content.

### STORES-001 — `restore()` loses entity IDs, breaking every restored entity's provenance chain
**File:** `src/ops/backup.ts:114`
**Description:** `restore()` calls `stores.canonical.create({kind, name, attributes})` (no `id` passed) — the local store unconditionally assigns `id: randomUUID()` (`local-canonical-store.ts:51`). So every restored entity gets a NEW id. But provenance events restored 60 lines later (line 172) still carry the ORIGINAL `subjectId` from the backup → every restored entity's provenance chain is broken. `TraceBuilder.traceEntity` will flag every restored entity as "missing provenance." `verify()` (lines 96–105) will flag every restored event as `orphan_subject`. None of this is caught by Phase 12 tests — Proof 2 verifies only artifact-ID preservation, and Proof 12 calls `doctor()` (reachability) instead of `verify()`.
**Recommendation:** Mirror the artifact-side pattern: add an `importSnapshot(entity)` method to `CanonicalStore` that preserves the original `id`/`createdAt`/`updatedAt`, or add a `createWithId` overload. `restore()` should use it instead of `create()`. Add a Phase 12 proof: assert restored entity ids match originals AND `verify()` returns `healthy` after restore.
**Evidence:** `backup.ts:110` `const exists = await stores.canonical.exists(entity.id);` — note: the check uses the ORIGINAL id. `backup.ts:114` `await stores.canonical.create({kind: entity.kind, name: entity.name, attributes: entity.attributes});` — note: no `id` field. `local-canonical-store.ts:51` `id: randomUUID()`. `test/phase12-proof.test.ts:90` (Proof 2) tests only artifact ids.

### SURFACE-001 — `cluster_resolve` MCP tool returns raw artifact objects (storagePath leak)
**File:** `src/mcp/server.ts:376`
**Description:** The `cluster_resolve` MCP tool does NOT apply `sanitizeArtifactForOutput` to artifact URIs. When a caller passes an artifact URI (e.g. `cluster://artifact/<id>`), the handler returns `object: resolved.object` raw — including `storagePath`, the absolute filesystem path `.db-cluster/content/<contentHash>`. Any MCP host running in the same filesystem context (the default for local stdio MCP servers) can `readFileSync(storagePath)` to obtain raw artifact content. This is an explicit bypass of the Phase 6 "artifact content boundary" claim documented in the server header (lines 590–597). `sanitizeArtifactForOutput`'s own JSDoc advertises this as the content escape hatch: "Full content available via cluster_resolve on the artifact URI." Combined with `cluster_find_sources` (which gives the AI any matching artifact URI), an AI agent has a two-step recipe to escape the content boundary on every artifact.
**Recommendation:** Apply `sanitizeArtifactForOutput` to `cluster_resolve` results when `resolved.store === 'artifact'`. Stop documenting `cluster_resolve` as the content escape hatch. Strip `storagePath` from every artifact returned by every MCP tool. Wire the SDK through `PolicyEnforcedKernel` (see SURFACE-002) so the policy layer's `redactArtifact()` actually runs at runtime.
**Evidence:** `src/mcp/server.ts:376-384` — case `cluster_resolve` returns `object: resolved.object` with no sanitization. `src/policy/redactor.ts:35-36` confirms `storagePath` is treated as sensitive by the policy layer. `src/mcp/server.ts:599-609` — `sanitizeArtifactForOutput` strips only `content`/`rawContent` (fields that don't exist on `Artifact` per `src/types/artifact.ts`), leaving `storagePath`. `src/adapters/local/local-artifact-store.ts:79` — `storagePath: join(this.contentDir, contentHash)` confirms it's the real local path.

### SURFACE-002 — SDK, CLI, and MCP never instantiate `PolicyEnforcedKernel`
**File:** `src/sdk/cluster-sdk.ts:82`
**Description:** The SDK constructs raw `new ClusterKernel(stores, ...)` instead of `PolicyEnforcedKernel`. The MCP server delegates to the SDK (`src/mcp/server.ts:14-19`). The CLI does the same (`src/cli.ts:15-22`). Every read AND write through any product surface bypasses policy enforcement entirely. The SDK accepts `options.policies`/`options.trustZones`/`options.visibilityRules` but only uses them inside `policyExplain`/`policyTest` for dry-run explanation. Actual kernel calls (`findSources`, `retrieveBundle`, `resolve`, `traceObject`, `proposeMutation`, `commitMutation`) never consult these options. **Configuring policies on the SDK is a no-op for everything except `policyExplain`/`policyTest`.** Combined with SURFACE-001, this means `storagePath` leaks unredacted on every MCP/SDK read path because `PolicyEnforcedKernel.redactArtifact` never runs. The wave5-redaction tests pass only by constructing `PolicyEnforcedKernel` themselves — giving false confidence.
**Recommendation:** Switch `ClusterSDK` to instantiate `PolicyEnforcedKernel` when `options.policies` is provided, and require a `Principal` in the SDK constructor (or per-call). Wire the CLI to construct `PolicyEnforcedKernel` from a `Principal` derived from auth / OS user / env. For the MCP server, accept a `Principal` from `DB_CLUSTER_PRINCIPAL` env or via a JSON-RPC handshake and pass it through.
**Evidence:** `src/sdk/cluster-sdk.ts:80-88` — `this.kernel = new ClusterKernel(stores, ...)` (NOT `PolicyEnforcedKernel`). `src/cli.ts:15-22` — same. `src/mcp/server.ts:14-19` — `getSDK()` constructs `ClusterSDK` without policies. `src/kernel/policy-enforced-kernel.ts:68-84` — `PolicyEnforcedKernel` exists and exposes the redacted wrappers but is never instantiated by any product surface.

### TESTS-001 — "No mutation without command" negative test is structurally trivial
**File:** `test/proof.test.ts:84`
**Description:** The "kernel does not expose raw store write access" test enumerates kernel **prototype** properties looking for `'stores'`/`'canonical'` etc. Constructor-injected fields like `private readonly stores: ClusterStores` do NOT live on the prototype — they live on instances. The test is trivially green regardless of whether the kernel actually hides stores. In JavaScript, TypeScript's `private` is a compile-time fiction; `(kernel as any).stores.canonical.create({...})` works at runtime and bypasses the entire command lifecycle. The test claims to prove "No mutation without command" but proves nothing about it. The only real drift-detection negative test (phase8-proof Proof 3 — direct adapter write → no receipt) is Postgres-gated and currently skipped.
**Recommendation:** Replace prototype check with an actual escape attempt. Best fix: hide `stores` behind a `WeakMap` or closure so `(kernel as any).stores` is genuinely `undefined` at runtime — then `expect((kernel as any).stores).toBeUndefined()` becomes meaningful. Also port phase8-proof Proof 3 to the local adapter so the drift-detection invariant has always-on coverage (see TESTS-008).
**Evidence:** `proof.test.ts:84-96` — `Object.getOwnPropertyNames(Object.getPrototypeOf(kernel)).filter((k) => k !== 'constructor')` — prototype names never include constructor params regardless. `src/kernel/cluster-kernel.ts:78` has `private readonly stores: ClusterStores` as a constructor parameter — TS-private only.

### TESTS-002 — "SQL is parameterized" has zero negative tests
**File:** `test/(none)`
**Description:** No test ingests `'; DROP TABLE canonical_entities; --` style payloads into entity kind/name/attributes/ID and asserts they're treated as data. `PostgresCanonicalStore` IS correctly parameterized today (all queries use `$N` placeholders), but a future PR that introduces string interpolation in any new query would not be caught. For an "AI-native federated database cluster" positioned at safety, this is a missing safety net.

Gate on Postgres availability the same way `phase8-proof` is gated.
**Evidence:** Grep for `DROP TABLE|OR 1=1|injection` across `test/` returns no matches. Postgres canonical store is the only SQL backend.

### TESTS-003 — "Artifact storagePath is sandboxed" has zero negative tests
**File:** `test/(none)`
**Description:** No test attempts to ingest an artifact with `filename = '../../etc/passwd'` or `contentHash` with traversal characters and asserts the write stays within the cluster directory. `LocalArtifactStore.ingest()` is safe today (`storagePath = join(contentDir, contentHash)` where `contentHash` is sha256), but `LocalArtifactStore.importSnapshot` (`local-artifact-store.ts:105`) accepts caller-supplied `metadata: Artifact` and uses `metadata.contentHash` to construct the path — a tampered backup with `contentHash` like `'../../escape/file'` would write OUTSIDE the contentDir.
**Recommendation:** Add a test in `test/restore-artifacts.test.ts` that mutates a backup's `artifactSnapshots[0].metadata.contentHash` to `'../../../escape-path-test'` and asserts restore either refuses it OR proves the write stayed within the artifact `contentDir`. Add a test that ingests an artifact with filename `'../../escape.txt'` and asserts only `metadata.filename` stores the literal string — no filesystem write occurs at that path.
**Evidence:** `local-artifact-store.ts:107` — `const contentPath = join(this.contentDir, metadata.contentHash);` with no defensive `realpath().startsWith(this.contentDir)` check.

### KERNEL-002 — Helper write paths build synthetic commands that never enter the CommandQueue
**File:** `src/kernel/cluster-kernel.ts:103` (`ingestArtifact`), `:160` (`createEntity`), `:212` (`linkEvidence`), `:603` (`rebuildIndex`)
**Description:** All four manufacture a `Command` via `proposeCommand → validateCommand → markCommitted` inline without ever calling `this.saveCommand()` and without entering the CommandQueue. The receipt's `commandId` therefore references a command that doesn't exist in any inspectable surface — `inspectCommand(receipt.commandId)` throws `NotFoundError`. Synthetic commands also never pass through `approveMutation`, so trust-zone approval gates do not apply. Soft bypass of "every mutation has an inspectable typed command lifecycle" — receipts cite phantom commands.
**Recommendation:** Either (1) call `this.saveCommand(cmd)` after manufacturing the synthetic command so `inspectCommand` can resolve it, or (2) refactor these helpers to delegate to `proposeMutation → commitMutation` so they share the queue/approval/policy path. Add a regression test: for every helper-method receipt, `kernel.inspectCommand(receipt.commandId)` must succeed.
**Evidence:** `cluster-kernel.ts:139-144` (ingestArtifact) `const cmd = markCommitted(validateCommand(proposeCommand(...)))` — no `saveCommand` call. Same pattern at lines `192-196` (createEntity), `239-247` (linkEvidence), `649-653` (rebuildIndex).

### KERNEL-004 — `PolicyEnforcedKernel.retrieveBundle` has no per-entity policy filtering
**File:** `src/kernel/policy-enforced-kernel.ts:195`
**Description:** `retrieveBundle` applies a single blanket `read_derivative` `enforce()` and a single blanket `collectRedactionRules()` for the whole bundle. It does NOT do per-entity / per-artifact policy filtering like `findSources` does. So if the principal can `read_derivative` on the index but is denied `read_owner_truth` on a specific entity, the full Entity object (or only blanket-redacted) is still returned inside the bundle. Compared to `findSources`'s per-object filter, `retrieveBundle` is the weaker enforcement path for the same underlying data.
**Recommendation:** Mirror `findSources`' per-entity / per-artifact check inside `retrieveBundle` (or refactor both to call a shared helper). For each resolved entity, `evaluatePolicy(read_owner_truth, ownerStore=canonical, resourceUri, entityKind)` and drop denied entities. Apply per-entity redaction rules collected from each entity's decision, not a single blanket decision.
**Evidence:** `policy-enforced-kernel.ts:195-219` — `enforce` called once at line 196, then `.map((re) => ({ ...re, object: redactEntity(re.object, rules) }))` applies only bundle-level rules. No per-entity `evaluatePolicy`. Contrast with `findSources` lines `151-186`.

### KERNEL-005 — No try/catch around post-mutation provenance + receipt → orphan-mutation window
**File:** `src/kernel/cluster-kernel.ts:321`
**Description:** `commitMutation`, `ingestArtifact`, `createEntity`, `linkEvidence`, `rebuildIndex`, and `compensateMutation` all execute store mutations and then call `recordProvenance` + `emitReceipt` with NO try/catch around the post-mutation steps. If the ledger fails after the canonical/artifact mutation has succeeded (network blip, ledger backend full, transient write error), the store retains the change but no receipt is emitted and no `mutation_committed` provenance event exists. In `commitMutation` specifically (line 428), `this.saveCommand(committed)` runs BEFORE `recordProvenance` — so the command is persisted as `committed` but has no associated receipt. Direct bypass path for "every committed command produces a receipt."
**Recommendation:** Wrap post-mutation steps in try/catch. On failure, either (a) attempt compensating writes to roll back the store mutation, or (b) emit a `mutation_orphaned` ledger event and surface the failure as a typed `ReceiptFailedError`. Add a test that injects a ledger failure mid-commit and asserts either rollback or orphan-event creation.
**Evidence:** `cluster-kernel.ts:354-447` — no try/catch wrapping lines 360–449; the only try/catch is at lines 339–345 (validation). Same lack of error handling at `cluster-kernel.ts:110-152` (ingestArtifact), `167-205` (createEntity), `217-257` (linkEvidence), `605-661` (rebuildIndex), `538-579` (compensateMutation).

### KERNEL-006 — `commitMutation` accepts `status='proposed'` → approval gate is skippable
**File:** `src/kernel/cluster-kernel.ts:328`
**Description:** Line 328: `const committableStatuses = ['proposed', 'validated', 'approved'];`. A caller can skip the `approveCommand` gate entirely: propose → commit with validation done inline. The "proposed → validated → approved → committed" doctrine is not enforced at the kernel layer. Trust-zone approval is enforced only in `PolicyEnforcedKernel.commitMutation` (via `enforce('commit_command')`) — so any path holding a `ClusterKernel` reference can commit without approval. Combined with KERNEL-001 (7 verbs unwrapped) and SURFACE-002 (CLI/SDK/MCP never instantiate `PolicyEnforcedKernel`), the approval gate is currently universally bypassable in production.
**Recommendation:** Either (a) tighten `commitMutation` to require `status='validated' || 'approved'`; (b) introduce a `KernelOptions.requireApproval: boolean` defaulting to true that enforces `status==='approved'`; (c) document explicitly that approval is policy-layer only and that `ClusterKernel` direct usage is for trusted internal/test code only. (b) is safest and most defensible.
**Evidence:** `cluster-kernel.ts:328`. `PolicyEnforcedKernel.commitMutation` (`policy-enforced-kernel.ts:280-283`) calls `enforce('commit_command')` — the only place approval is gated.

### KERNEL-007 — `commitMutation` switch arm for `link_evidence` is a NO-OP
**File:** `src/kernel/cluster-kernel.ts:406`
**Description:** The `link_evidence` case (lines 406–414) pushes IDs and writes a `resultSummary` string but never calls `this.stores.ledger.append`, never verifies artifact/entity exist, never produces a provenance link event. The kernel's helper method `linkEvidence` (lines 212–258) DOES verify existence and append a provenance event — but the `commitMutation` switch path does NOT. **Evidence:** `cluster-kernel.ts:406-414`. Contrast with `cluster-kernel.ts:217-237` (helper). `trace-builder.ts:125` matches on `action === 'evidence_linked'`.

### STORES-002 — Event/receipt restore is non-idempotent (silent duplication)
**File:** `src/ops/backup.ts:166`
**Description:** Restore's skip check at line 168 reads `ledger.getEvent(event.id)` (the ORIGINAL id), but `ledger.append()` at line 172 unconditionally generates a fresh `randomUUID()` and timestamp (`local-ledger-store.ts:30-32`). **Evidence:** `backup.ts:168-181` — `getEvent(event.id)` check vs `ledger.append({...})` insert (no id passed). `local-ledger-store.ts:30` — `id: randomUUID()`.

### STORES-004 — Tokenizer breaks for any non-ASCII content (content indexer silently empty)

### STORES-005 — Local stores have no file locking and use non-atomic writes (silent data loss + corruption on crash)
**File:** `src/adapters/local/local-canonical-store.ts:89` (and `local-artifact-store.ts:132`, `local-index-store.ts:88`, `local-ledger-store.ts:112`)
**Description:** All four local stores persist via plain `writeFileSync` to a single JSON file with no locking, no atomic rename, no fsync. (1) **Concurrent writers** — if two kernel instances point at the same dataDir, both `load()`, both mutate independent maps, both `writeFileSync` — last-writer wins, silent data loss. (2) **Crash mid-write** — `writeFileSync` is not atomic. ## 8. Cross-cutting themes

1. **Policy/redaction layer is correct in isolation but unwired in production.** `PolicyEnforcedKernel` exists, `redactor.ts` works, `wave5-redaction.test.ts` proves both. But SDK, CLI, and MCP all instantiate raw `ClusterKernel` (SURFACE-002). 7 ClusterKernel verbs have no wrapper anyway (KERNEL-001). `matchStores` default-allows on underspecified requests (SURFACE-004). `findSources`/`retrieveBundle`/`explainIndex` leak data even when the wrapper IS used (KERNEL-003/-004/-011). `ClusterKernel` is exported as public API (KERNEL-013) so any consumer can grab it directly. **The entire policy story is currently theater.**

2. **`restore()` is broken in multiple ways and tests don't catch it.** Entity IDs lost (STORES-001) → provenance chains shredded. Events non-idempotent (STORES-002) → silent duplication on re-run. Fallback path silently loses artifact IDs (STORES-003). Phase 12 Proof 12 calls `doctor()` (reachability) instead of `verify()` (orphan-subject detection) — so the regression is invisible (STORES-009). Phase 12 was self-declared PASS but it shipped a backup/restore that fails its own integrity claims under any thorough verification.

3. **The release-readiness story has no CI.** No `.github/` at all (CIDOCS-001). The release-gate script uses Windows-only `findstr` and silently passes on macOS/Linux (CIDOCS-003). Examples never typecheck (CIDOCS-007) so they can rot silently (CIDOCS-002 — every example is currently broken). `prepublishOnly` is not set (CIDOCS-011) so `npm publish` could skip everything. The "release-ready" claim in README, phase-15-closeout, and release-readiness.md is unsupported by any continuous verification.

4. **Safety invariants lack adversarial test coverage.** SQL injection (TESTS-002) → zero negative tests. Path traversal sandbox (TESTS-003) → zero negative tests. "No mutation without command" (TESTS-001) → the only headline test is structurally trivial. Drift detection (TESTS-008) → only in Postgres-gated suite. The code is mostly safe today, but the safety net is mostly aspirational.

5. **Commit-path divergence from helper-path.** The 4 helper write methods (`ingestArtifact`/`createEntity`/`linkEvidence`/`rebuildIndex`) and the `commitMutation` switch arms for the same verbs do DIFFERENT work. `commitMutation`'s `link_evidence` arm is a NO-OP (KERNEL-007). `commitMutation`'s `ingest_artifact` arm doesn't index (KERNEL-008). Helpers bypass the CommandQueue (KERNEL-002). Two incomplete write surfaces with subtly different semantics — and the docs/tests don't distinguish them.

6. **Local stores have no file locking + non-atomic writes.** Concurrent writers race (STORES-005). Crash mid-write corrupts JSON (STORES-005 + KERNEL-021). `load()` has no recovery — throws into the constructor. Two `db-cluster` CLI invocations against the same cluster will silently lose data. CommandQueue has the same fragility (KERNEL-021).

7. **Fixture hygiene smells across the test suite.** `restore-artifacts`, `command-persistence`, all `dogfood-*` files, `content-index` use `beforeAll`/`afterAll` instead of `beforeEach`/`afterEach` (TESTS-007). State leaks between `it()` blocks. Reordering tests could break or stop testing what they claim. Multiple "either-OK" status assertions (TESTS-018). Several tests with description/body mismatches (TESTS-012, TESTS-021).

8. **Dashboard "redaction" is misleadingly named.** `dashboard/components/PolicyViewToggle.jsx` has an `applyRedaction` function that's a client-side cosmetic filter (SURFACE-012). The test file maintains its OWN copy of it instead of testing the JSX component (TESTS-004). Together: a security-relevant boundary that doesn't actually enforce security and isn't actually tested. The README footer even claims "Redaction is a view-layer filter" — which is true but invites the wrong mental model.

9. **The `repo-knowledge` integration is internally inconsistent.** `update-workflow.ts` correctly threads everything through `propose → validate → approve → commit`. `ingest.ts` bypasses the lifecycle entirely with direct kernel calls (SURFACE-003). Two adjacent files modeling opposite patterns. The architecture doctrine has a built-in counterexample in its own codebase.

10. **Pagination / "limit:100000" magic numbers are pervasive.** `indexStatus`, `listStaleRecords`, `update_entity` rebuild use the `{text:'', limit:100000}` pattern that isn't in the contract and silently caps results (KERNEL-012). `traceForwardIndex` uses `limit:100` (STORES-012). `TraceBuilder.traceArtifact` does unfiltered `listEvents()` per node (KERNEL-015). The contract needs `listAll`/`findBySourceId`/`countByX`/paginated iteration to fix this class.

## 10. Per-domain summaries (verbatim)

### Kernel
Kernel domain is structurally well-organized — store boundaries are clean, types are coherent, command lifecycle helpers exist as pure functions in `commands.ts`, and the CommandQueue rewrite for cross-instance freshness is correctly implemented (no in-memory caching that would defeat it). However, the audit surfaces THREE critical/high issues that directly bypass cluster doctrine: (1) **KERNEL-001** — seven `ClusterKernel` methods including `ingestArtifact`, `createEntity`, `linkEvidence` have NO `PolicyEnforcedKernel` wrapper, allowing full unredacted writes to bypass policy if a caller holds a `ClusterKernel` reference (and the public API actually exports `ClusterKernel`, see KERNEL-013); (2) **KERNEL-003** — `PolicyEnforcedKernel.findSources` returns `indexRecords` unfiltered, leaking restricted entity kind/name/attributes via the discovery surface even when the entity is policy-filtered; (3) **KERNEL-007 / KERNEL-008** — the `commitMutation` switch arms for `link_evidence` and `ingest_artifact` are functionally broken (`link_evidence` is a NO-OP that emits no link, `ingest_artifact` never indexes the artifact), creating divergence between the helper-method write path and the propose→commit write path. Secondary concerns: receipt completeness has a real bypass (no try/catch around the post-mutation provenance+receipt sequence — KERNEL-005), command lifecycle bypass (proposed→committed allowed without approval — KERNEL-006), `markRejected` is unguarded and loses actor+reason (KERNEL-009). Test coverage gap (KERNEL-014) is the meta-finding: no verb-parity test exists between `ClusterKernel` and `PolicyEnforcedKernel`, which is exactly how KERNEL-001 escaped review. Confidence is LOW on the postgres-adapter side (not in scope) — the empty-text/limit:100000 patterns may behave very differently on postgres; the local-adapter tests pass which masks contract-level brittleness (KERNEL-012). Overall: kernel doctrine is mostly enforced for the propose→approve→commit path but has multiple bypass routes via helper methods, the `@internal _kernel` getter, and direct `ClusterKernel` imports.

### Stores
Stores layer has solid surface-level invariants (SQL is fully parameterized, artifact `storagePath` uses content-hashed paths, index is consistently treated as derivative by retrieval/rebuild/verify) but has serious correctness gaps in the restore path that masquerade as fixed. The biggest concern is **STORES-001**: restored entities lose their original UUIDs and the subsequently-restored provenance events still carry the original subject ids, silently shredding every entity's provenance chain after restore. Phase 12 Proof 12 calls `doctor()` (reachability only) instead of `verify()`, so the regression is invisible to the test suite. **STORES-002** (non-idempotent event/receipt restore) and **STORES-003** (silent ID loss in importSnapshot fallback) compound this — restore is far less safe than the Phase 12 narrative claims. Tokenizer unicode breakage (**STORES-004**) silently neuters content-aware indexing for any non-English file. Lower-confidence areas: I did NOT run the test suite, did NOT exercise the cluster with concurrent writers to confirm the data-loss prediction in **STORES-005**, and did NOT load a malicious backup to confirm the path-traversal mitigation chain in **STORES-006** — these are reasoned from static reading. SQL injection and storagePath sandbox audits are high-confidence.

### Surface
Surface health: the architecture is well-designed and the kernel + policy + redactor layers are correct in isolation — but the product-facing surfaces (CLI, SDK, MCP server) never instantiate `PolicyEnforcedKernel`, so the entire policy/redaction system is effectively unwired in production. Top three concerns: (1) `cluster_resolve` MCP tool returns raw artifact objects with no sanitization, including `storagePath` (the absolute filesystem path) — an AI can use this as a documented content-boundary escape; this same `storagePath` leak also affects `find_sources`/`retrieve_bundle` which only strip `content`/`rawContent` (fields that don't even exist on `Artifact`). (2) SDK + CLI + MCP all wire raw `ClusterKernel`, not `PolicyEnforcedKernel`, so artifact content boundary, store-scoped denies, and redaction rules never fire at runtime — wave5-redaction tests pass only because they construct `PolicyEnforcedKernel` themselves. (3) The repo-knowledge ingest integration calls `kernel.createEntity` / `ingestArtifact` / `linkEvidence` directly, bypassing the propose→validate→approve→commit command lifecycle, contradicting the sibling `update-workflow.ts` which models the correct pattern. Lower confidence: prototype-pollution probes against `args.payload` appear safe under Node 22 spread semantics but were not exhaustively fuzzed; the MCP `Server` class is documented as not enforcing inputSchema in the cases I read, but I did not unit-test malformed-input behavior of every tool. Best evidence the audit is grounded: `wave5-redaction.test.ts` proves the redactor works AND proves it's tested only via direct `PolicyEnforcedKernel` use; SURFACE-001's bypass is documented in the very comment block (`_contentAccess: 'Full content available via cluster_resolve...'`) that purports to describe the content boundary.

### Tests
The test suite has good breadth (614 passing across 51 files) and is genuinely load-bearing for the happy paths — `phaseN-proof` files exercise real kernel/SDK/MCP paths, not mocks. The negative-test coverage matrix is uneven: invariants 2 (index derivative), 4 (artifact content boundary), 8 (receipt completeness), 9 (lifecycle transitions), and 10 (cross-instance freshness) have strong negative tests; invariants 1 (no mutation without command), 3 (redaction on every read), 6 (artifact storagePath sandbox), 7 (backup integrity) have partial or trivially-passing negative tests; invariants 5 (SQL parameterization) and 6 (path sandbox) have ZERO adversarial tests. Top 3 concerns: (1) **TESTS-001** — the headline "No mutation without command" negative test is structurally trivial (checks prototype names that wouldn't be there anyway); the only real drift-detection proof is Postgres-gated and currently skipped. (2) **TESTS-002/003** — load-bearing safety invariants (SQL injection resistance, path sandbox) have NO adversarial tests at all; the code is safe today but has no regression net. (3) **TESTS-004** — `dashboard-policy-view.test.ts` tests a local copy of `applyRedaction` instead of the real `PolicyViewToggle.jsx` implementation, giving false confidence on a UI security boundary. Weakest negative coverage: invariants 5 (SQL parameterization — zero coverage) and 6 (storagePath sandbox — zero coverage); invariant 1 has only structural coverage that doesn't actually probe the failure mode. 48 skipped tests are all 4 Postgres-gated files (`backend-parity`, `phase8-proof`, `postgres-canonical-store`, `postgres-kernel-regression`); the drift-detection invariant in `phase8-proof` Proof 3 has no local equivalent. Fixture hygiene smell: several files use `beforeAll`/`afterAll` instead of `beforeEach` (`restore-artifacts`, `command-persistence`, all `dogfood-*`, `content-index`) creating order-dependent tests. Two portability bugs in `install-smoke.test.ts` (`Remove-Item` PowerShell cmdlet, `type` Windows command).

### CI/Docs
Overall ci/docs health is MIXED: the documentation infrastructure is unusually thorough (handbook, package-boundary, release-readiness, 15 phase-closeouts, full CLI/SDK/MCP refs) and the test coverage is genuine (614/614 passing). But three load-bearing claims that gate the "Release Readiness PASS" verdict are not actually true: (1) there is no `.github/` directory at all, so "release-ready" is dead-letter — no CI continuously verifies the build/test/smoke claims [**CIDOCS-001** HIGH]; (2) every TypeScript example is broken at the public-API level (missing required `actorId`, wrong PolicyEnforcedKernel constructor, wrong findSources signature, references to renamed methods `validateCommand`/`approveCommand`, plus one example that violates the documented package-boundary by importing from `'../../src/'`) — directly contradicting `docs/phase-10-closeout.md` "All compile" and `docs/release-readiness.md` "Examples use package imports ✓" [**CIDOCS-002** HIGH]; (3) the release-gate's docs-drift check uses Windows-only `findstr` and silently passes on macOS/Linux, AND the gate is not invoked by any CI — so even the manual gate doesn't catch the drift that exists right now [**CIDOCS-003** HIGH]. Secondary concerns: `docs/sdk.md` and `docs/cluster-uris.md` document an SDK constructor signature that doesn't exist, including a Postgres-via-SDK pattern that's structurally impossible [CIDOCS-005]; `tsconfig` only typechecks `src/**/*`, leaving examples and scripts to rot [CIDOCS-007]; three loose research `.txt` files at repo root hurt the "release-ready" signal on GitHub [CIDOCS-009]; `.db-cluster/` is missing from `.gitignore` though `docs/quickstart.md` tells users to run init in cwd [CIDOCS-010]. Top 3 concerns: no CI (CIDOCS-001), broken examples (CIDOCS-002), non-portable release-gate (CIDOCS-003). Lowest confidence: I did not exhaustively walk every phase-N-closeout for stale code references (spot-checked Phases 10-15); the dogfood scripts (`scripts/dogfood-*.ts`) were not deeply read since they're internal-only per `files` field, though I confirmed they import from `../src/` which is fine inside-tree but invisible to npm consumers.
