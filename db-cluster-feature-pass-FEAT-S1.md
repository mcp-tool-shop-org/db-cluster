# db-cluster v2.0 — Consolidated Feature-Pass Report

**Thesis anchor:** db-cluster exists so AI agents get *trustworthy retrieval + a safe mutation lifecycle*, and operators get *safe writes + real diagnostics*. Every judgment below is scored against that one sentence.

**Dedupe note:** the raw audit returned 47 candidate features. One capability — **canonical version-history reads** — was filed independently by three domains (FEAT-AI-001 MCP, FEAT-SDK-001 SDK/kernel, FEAT-CLI-006 CLI). These are one kernel capability surfaced on three faces; they are merged into a single feature (**FEAT-VERSIONS-001**) below. Final post-dedupe set = **45 features**. No features were invented; no genuine feature was dropped. Semantic search (FEAT-RETR-002 substrate + FEAT-AI-008 MCP tool) and batch ops (FEAT-SDK-003 + FEAT-AI-005) are *related but distinct* layers and are kept separate with explicit cross-references and sequencing.

## 0. Phase-6 locked decisions (advisor sign-off)

Two scoping forks were routed to the advisor mid-synthesis and are now **locked**. They override the "pick one" framing in §4 below where they conflict.

**Storage lane → SQLite (full four-store), reframed as the *retrieval vehicle*, not a separate lane.**
- FTS5 *is* the ranked-full-text engine FEAT-RETR-001 needs — so SQLite does not spend scope *away* from retrieval, it makes the retrieval headline cheaper and more robust. Couple **FEAT-RETR-001 to FTS5 explicitly**.
- Keep the JSON file-store as a **zero-dep fallback** with basic unranked search; SQLite becomes the "real retrieval" local adapter.
- Execution detail (decide at build time): `node:sqlite` (built-in, Node 22+, may bump the engine floor — acceptable for a major) vs `better-sqlite3`.
- This **folds in** FEAT-STORE-004 (atomic ledger append), most of FEAT-RETR-007 (scale), FEAT-RETR-005 `fields`/`phrase`, and FEAT-STORE-011 (full-text) as a single workstream.

**Postgres parity (FEAT-STORE-001) → handled as a CLAIM-FIX, not a build.** Document Postgres as **canonical-only** and defer full artifact/index/ledger parity to **2.1** — same discipline as the retracted SSL claim. This closes the implied-parity integrity gap without crowding the headline. (Re-classify STORE-001 from "CRITICAL build" to "2.1 + a 2.0 doc reconciliation" in execution.)

**Recommended cut → confirmed ("Cut looks right").** v2.0 = **ranked full-text (RETR-001) + bundle scores/snippets (RETR-004) + version-history (VERSIONS-001) + the claim-fixes + SQLite**, carrying the security hardening. Two riders are now mandatory:
1. **Claim-fixes must explicitly re-scope "vector lookup"** in the `index-store` contract + README to **"ranked full-text now; semantic/vector tracked for 2.x"** — the honest counterpart to NOT rushing semantic search.
2. **Version-history ships *policed*** — a new gated read path through `PolicyEnforcedKernel` (capability check + redaction), never a raw exposure. (Already the hard dependency on FEAT-VERSIONS-001.)

**Semantic / vector retrieval (FEAT-RETR-002 + FEAT-AI-008) → explicitly DEFERRED to 2.1, study-swarm-gated.** It is the hard half (embedding-model choice, ANN indexing, hybrid lexical+vector ranking, eval methodology) and must NOT be improvised. Ship the *confident* half (FTS5-ranked) in 2.0; do semantic as a study-swarm-grounded 2.1 item. Hard sequence stays: **RETR-001 → RETR-002 → AI-008.**

## 2. Claim / reality mismatches

Ten features carry `category=claim-mismatch` — documented promises the code does not keep. These are the highest-trust-risk items for a *security-hardening* major: a 2.0 that ships with a false capability claim on a core contract reads as careless.

- **[FEAT-RETR-003]** — *the anchor.* The index-store contract advertises **"full-text/vector lookup"** but delivers neither: search is literal `text.toLowerCase().includes(query)` substring containment (no query-time tokenization, zero ranking), and `embedding?: number[]` is written/read nowhere. **Location:** `README.md:62`; `src/contracts/index-store.ts:5`; `docs/store-contracts.md:52` (+ 8 translated READMEs at line 62). Contrast `docs/retrieval-bundles.md:3`, which already states the truth — the repo contradicts itself. **Implement-or-retract:** ship FEAT-RETR-001 (real BM25) so "full-text (ranked)" becomes true, then **drop the word "vector"** until FEAT-RETR-002 lands; regenerate translations before tag/publish. If neither implementation lands, retract both words now.

- **[FEAT-AI-006]** — Two on-wire MCP error shapes coexist. The documented agent recipe `JSON.parse(result.content[0].text)` matches the catch arm, but the two INJECT-001 gate refusals (commit + compensate — the *most important* errors in the safe mutation lifecycle) return a bare spread body with no `isError`/`content[]` wrapper, so an agent following the docs calls `JSON.parse(undefined)` on exactly those refusals. **Location:** `docs/mcp.md:227-243, 257-261` vs `src/mcp/server.ts:1014-1018, 1078-1082`. **Implement-or-retract (breaking, permitted):** route the gates through the documented wrapper — best, have them throw a typed `PolicyDeniedError` so the single catch-arm path produces the one documented shape.

- **[FEAT-AI-007]** — ListTools emits a **private 5-field** annotation vocabulary (`readOnly/writesCluster/approvalSensitive/...`) while the docs promise the **spec-standard** `readOnlyHint/destructiveHint/idempotentHint`. No generic MCP host can auto-gate destructive tools or label read-only ones. **Location:** `docs/mcp.md:136-153` vs `src/mcp/server.ts:352-358, 1241-1249`. **Implement-or-retract (breaking, permitted):** emit BOTH — the spec hints under `annotations` (derived from the internal flags) on all 16 tools, plus the rich flags under a namespaced key. Update the docs table to names actually emitted.

- **[FEAT-CLI-001]** — *highest-leverage operator fix in the build.* `verify`/`doctor`/`health` report corruption but **exit 0**, silently passing CI/cron. Docs promise "0 on success, 1 on any failure." **Location:** `docs/cli.md:57` vs `src/cli.ts:316-337, 341-362, 434-461`. **Implement:** three-line `if (!result.ok) process.exitCode = 1;` per handler. Then retract the `operations.md:58` "OR ok:false" workaround the doc author added precisely because the exit code was broken.

- **[FEAT-CLI-002]** — Documented `--version`/`-v` falls through to help. **Location:** `docs/cli.md:60-62`. **Implement:** print `package.json` version (plain + `{version}` under `--json`). Every CLI ships `--version`; retracting is the wrong call.

- **[FEAT-CLI-003]** — Docs promise color auto-disables on non-TTY; code only disables on explicit flags, so `get id > file` embeds raw ANSI. **Location:** `docs/cli.md:68` vs `src/cli/color-output.ts`. **Implement:** gate on `process.stdout.isTTY && !NO_COLOR`, honor `FORCE_COLOR`.

- **[FEAT-CLI-008]** — `--json` error envelope is `{ error: "free text" }` with no `code` field and binary 0/1 exit, so scripts cannot branch on error *type*. `OPS_ERRORS` codes exist but are unused. **Location:** `src/ops/errors.ts:25-32` (+ `cli.ts` `fail()`). **Implement (breaking, permitted):** emit `{ error: { code, message, hint } }` + a documented exit-code map.

- **[FEAT-SDK-008]** — If `docs/package-boundary.md` asserts the safe surface is sufficient, that promise is broken while version-history (and likely lineage/receipt reads) are `/unsafe`-only. **Location:** `docs/package-boundary.md` — *line unverified* (D3 had suppressed tool output; synthesis must locate the exact "safe surface covers X" assertion). **Implement-or-retract:** largely *discharged* by shipping FEAT-VERSIONS-001 + FEAT-SDK-009; then re-audit the boundary doc and either confirm completeness or document the intentional `/unsafe`-only operations.

- **[FEAT-STORE-002]** — README describes a **runnable "viewer-only React dashboard demo" in `dashboard/`**; no `.tsx/.jsx` exists anywhere in the repo (the lone `dashboard/apply-redaction.d.ts` header itself says the components "are not present"). View-*models* exist in `src/dashboard/*.ts`; the UI does not. **Location:** `README.md:77-83`. **Implement-or-retract:** cheapest honest path for 2.0 = **retract** (reword to "view-models live in `src/dashboard/`; no reference UI is bundled," delete the orphan `.d.ts`). A real single-file viewer is v2.x polish, not a 2.0 blocker.

- **[FEAT-STORE-011]** — `store-contracts.md` promises the index is "pluggable for richer backends (full-text, vector) later," but `IndexStore` has no pluggability seam and the factory hardcodes `LocalIndexStore`. **Location:** `docs/store-contracts.md:35-36`. **Implement-or-retract (partial):** if SQLite (FEAT-STORE-006) ships, FTS5 makes "full-text later" true; otherwise soften/defer the "vector" word.

**Context note (not a current mismatch):** Postgres **TLS** was honestly *retracted* in the security pass (`SECURITY.md:21-33`) — the docs are accurate today. It therefore appears below as a **missing-feature** (FEAT-STORE-003), not a claim-mismatch.

### CRITICAL

### [FEAT-VERSIONS-001] Gated, redaction-bound canonical version-history reads across kernel + SDK + MCP + CLI *(merged from FEAT-AI-001, FEAT-SDK-001, FEAT-CLI-006)*
- Priority: CRITICAL
- Category: missing-feature
- Thesis-fit: AI agent + operator. The hardening shipped canonical versioning (`update()` appends an immutable version; `create()` stamps v1) and `listVersions`/`getVersion` exist on the `CanonicalStore` contract + both adapters + the verify diagnostic — but there is **no policed path to read history** on `ClusterKernelInterface`, `PolicyEnforcedKernel`, SDK, MCP, or CLI. Today "what did this entity look like before / who changed it / show me v3" forces an agent to hand-walk provenance edges, and forces an *app* into `/unsafe` (bypassing the gate AND redaction) — the thesis inverted: the read that most needs provenance discipline is the one that escapes it. Removes that friction on all four surfaces at once.
- Scope: `src/contracts/canonical-store.ts` (signatures already present — no change); `src/kernel/cluster-kernel-interface.ts` + `cluster-kernel.ts` + `policy-enforced-kernel.ts` (add gated `listEntityVersions(uri)/getEntityVersion(uri,version)` — adding to the interface forces the gated wrapper via compile-time parity); `src/sdk/cluster-sdk.ts`; `src/mcp/server.ts` (2 read-only tools `cluster_list_versions`/`cluster_get_version` + annotations); `src/mcp/sanitize.ts` (historical-entity sanitizer); `src/cli.ts` (`db-cluster versions <id> [--at <n>] [--json]`); `docs/mcp.md`. Effort: medium  Breaking: no
- Dependencies: **HARD** — each historical revision is owner-truth and MUST pass the **same capability check (`read_owner_truth`) + the identical redaction/`sanitizeEntityForOutput` pipeline** as live `resolve`/`getEntity`; never return raw store rows; if artifact history is ever added it must strip `storagePath`/`content`. Reading a prior version requires the same capability as the current version. Implement at the **kernel layer once**; surface on all three+ faces for parity. (This single SDK accessor also unblocks the CLI facet.)
- Recommendation: **Build for v2.0** — the clearest case of a legitimate flow forced into `/unsafe`, and the single biggest retrieval gap relative to the versioning the hardening shipped. MCP tools annotated `readOnly:true` with `empty_reason` when no history; SDK returns typed records + JSDoc; CLI wrapper is small once the accessor lands.

### [FEAT-RETR-001] Real ranked text retrieval (BM25/TF-IDF over the existing tokenizer)
- Priority: CRITICAL
- Category: missing-feature
- Thesis-fit: AI agent. The thesis core. `LocalIndexStore.search` does `r.text.toLowerCase().includes(query.toLowerCase())` on the *whole query string* (never tokenized at query time) and returns results in Map-insertion order — **zero relevance scoring anywhere**. A natural-language query returns nothing unless the exact phrase appears verbatim, or an arbitrarily-ordered pile the agent cannot trust. This is the biggest gap between the product's promise and its behavior.
- Scope: `src/adapters/local/local-index-store.ts` (query-time tokenization + scoring); `src/indexing/tokenizer.ts` (reuse `tokenize`/`extractKeyTerms`; add doc-length/TF table); `src/contracts/index-store.ts` + `src/types/index-record.ts` (result may gain `score`); `src/types/evidence-bundle.ts` + `src/retrieval/retrieval-planner.ts` (thread score — see FEAT-RETR-004). Effort: medium  Breaking: yes (result ordering + record/result shape)
- Dependencies: Tokenizer already exists and is solid (Unicode-aware, stop-words) — reuse at query time. Storing TF/doc-length implies an index-format bump + rebuild; `rebuildIndex` already does an atomic `replaceAll` swap, so migration = "rebuild the derivative index" (no owner-truth migration). Pairs with FEAT-RETR-004.
- Recommendation: **Build for v2.0 as the headline retrieval feature.** BM25 (k1≈1.2, b≈0.75), pure-TS, zero new deps, fully local. Makes "full-text" TRUE (resolves half of FEAT-RETR-003) — the difference between a demo and a product an agent can ground on. *Sequencing:* defer vector (FEAT-RETR-002) to v2.x; ranked lexical is the 80/20 win and ships without an embedding dependency.

### [FEAT-CLI-001] Map diagnostic failures to non-zero exit codes (verify / doctor / health)
- Priority: CRITICAL
- Category: claim-mismatch
- Thesis-fit: Operator. "Real diagnostics" — a diagnostic that reports corruption but exits 0 silently passes CI/cron; the operator never learns the cluster is unsound. Makes `db-cluster verify && deploy` actually safe.
- Scope: `src/cli.ts` `handleVerify` (316-337), `handleDoctor` (341-362), `handleHealth` (434-461). Effort: small  Breaking: yes (scripts relying on a failing verify's exit 0 will start failing — correctly)
- Dependencies: none — result objects already carry `ok`/`status`; the CLI just fails to act on them.
- Recommendation: **v2.0.** `if (!result.ok) process.exitCode = 1;` per handler. Single highest-leverage operator fix in the domain — ship it regardless of what else makes the cut. Retract the `operations.md:58` "OR ok:false" workaround afterward.

### [FEAT-STORE-001] Postgres parity — artifact / index / ledger stores (a real federated cluster on Postgres)
- Priority: CRITICAL
- Category: missing-feature
- Thesis-fit: Operator. The thesis sells a production-grade *federated* cluster, but only the **canonical** store is durable on Postgres — artifact, index, and **ledger** silently fall back to single-node file adapters even when `backend=postgres`. The ledger is the provenance/receipt backbone, so the trust spine is the *least* scalable store, and a torn `events.json` under concurrent writers undermines "trustworthy provenance." Removes "I configured Postgres for durability but my event log still corrupts under parallel writers."
- Scope: `src/adapters/postgres/` (add `postgres-artifact-store.ts`, `postgres-index-store.ts`, `postgres-ledger-store.ts`); `src/adapters/postgres/schema.ts` + migrations 003/004/005; `src/adapters/factory.ts` (wire); `src/adapters/index.ts` (export). Artifact-content decision: bytea vs large-object vs filesystem-pointer. Effort: large  Breaking: no
- Dependencies: Needs the migration runner (FEAT-STORE-007) to be usable without hand-applied DDL. Ledger store must use a real serial/sequence for monotonic `seq`, not `array.length+1`. **Mutually exclusive storage-headline decision with FEAT-STORE-006 (SQLite) — pick one, not both.**
- Recommendation: **Build for v2.0 IF Postgres is the chosen storage headline.** Sequence ledger first (most write-heavy, most thesis-critical), then index, then artifact. bytea for small/medium blobs with a documented ceiling; leave the door open to an S3 pointer (FEAT-STORE-010) for large blobs. Honestly disclosed today (so not a claim-mismatch), but shipping "the v2.0 feature set" while the cluster is canonical-only on Postgres undercuts the "federated cluster" positioning.

### [FEAT-CLI-004] Progress feedback for long-running ops (--progress / streaming counters)
- Priority: HIGH
- Category: production-readiness
- Thesis-fit: Operator. A multi-minute rebuild/verify/backup that prints nothing is indistinguishable from a hang — on a 100k-entry store the operator waits blind, unable to tell slow from deadlocked. The seeded V2-C1-009 residual.

- Scope: `src/cli.ts` `handleMigrations`/`handleRestore`/`handleRebuild`. Prompt when interactive (TTY) unless `--yes`; always proceed under `--yes`/non-TTY for automation. Effort: small  Breaking: no

Low priority because v1 data-product operators are typically deliberate, but the asymmetry (irreversible migrate, backup-only recovery) makes it cheap insurance.

### [FEAT-STORE-005] Streaming artifact I/O for large blobs
- Priority: MEDIUM
- Category: performance
- Thesis-fit: App-dev / AI agent. The artifact store is for "raw immutable files, text, model outputs" and `store-contracts.md:23` claims "Large blobs supported" — but `getContent` reads the entire file into a Buffer and `put()` takes a fully-materialized Buffer/string. - Scope: `src/contracts/artifact-store.ts` (`putStream(Readable, mediaType)`, `getContentStream(id) → Readable`); `src/adapters/local/local-artifact-store.ts` (`createReadStream`/`createWriteStream` + streaming sha256 via a hash transform); `src/kernel/policy-enforced-kernel.ts:73-96` (`wrapArtifact` to gate + receipt the streaming paths). Effort: medium  Breaking: no (additive; keep buffer paths for small artifacts)
- Dependencies: Content-hash-as-id means hashing the stream as written — the id is known only after completion, so write-to-temp-then-rename for atomicity. Pairs with an S3 adapter (FEAT-STORE-010). *Overlaps FEAT-SDK-007 at the SDK layer.*

- Recommendation: **Add streaming as an additive contract method for v2.0** (keep buffer-based put/get for small artifacts). ### [FEAT-STORE-006] SQLite local adapter (concurrent, atomic alternative to the file adapter)
- Priority: MEDIUM
- Category: missing-feature
- Thesis-fit: App-dev / operator. A hard gap exists between the two current backends — local (single-node, non-atomic file rewrites, O(n) index/ledger) and Postgres (needs a server, canonical-only). A single-file SQLite adapter closes it: ACID transactions, real indexes, FTS5 for the index store, monotonic rowids for the ledger, WAL concurrency — zero server. It gives ALL FOUR stores parity in one adapter (unlike the partial Postgres path).
- Scope: new `src/adapters/sqlite/` (four stores + schema); `src/contracts/config.ts` (backend enum + `sqlitePath`); `src/adapters/factory.ts` (wire). FTS5 virtual table satisfies the index store's "pluggable richer backend" promise. Effort: large  Breaking: no
- Dependencies: Adds a native dep (`better-sqlite3` or `node:sqlite` on Node 22+; devDeps already target `@types/node ^22`). **Overlaps FEAT-STORE-004** (SQLite ledger fixes the append problem for free) and **FEAT-STORE-011** (FTS5 = real full-text). **Mutually exclusive storage-headline decision with FEAT-STORE-001.**
- Recommendation: **Strong candidate but scope-heavy — pick ONE of {complete Postgres parity (FEAT-STORE-001), SQLite adapter} as the v2.0 storage headline, not both.** Recommend **evaluating SQLite FIRST**: full four-store parity + atomic ledger + FTS5 search in one adapter with no server arguably serves the "embed a trustworthy cluster" thesis better than a partial Postgres. No docker-compose, no `.env` example, no `db-cluster migrate`. Real first-run friction for "safe writes + real diagnostics."

- Dependencies: **More important once FEAT-STORE-001 adds artifact/index/ledger tables.** Build the runner before or with parity, not after.

### [FEAT-STORE-008] Postgres connection resilience — pool config, timeouts, retry/reconnect
- Priority: MEDIUM
- Category: production-readiness
- Thesis-fit: Operator. The Pool is constructed with ONLY `{ connectionString }` — no `max`, no `idleTimeoutMillis`, no `connectionTimeoutMillis`, no statement timeout, no `pool.on('error')` handler (an unhandled idle-connection error can crash the process), and the config schema has nowhere to set them. For "durable, concurrent, production-grade," an operator cannot tune or harden the connection at all.
- Scope: `src/contracts/config.ts` (optional pool block: `max`, `idleTimeoutMillis`, `connectionTimeoutMillis`, `statement_timeout`); `src/adapters/factory.ts:44-57` (thread into Pool + attach `pool.on('error')`); optional bounded retry around transient query failures in `postgres-canonical-store.ts`. Effort: small  Breaking: no
- Dependencies: **Shares the config-surface + Pool-construction change with FEAT-STORE-003 (TLS) — do both in one pass.** Retry must respect the existing `FOR UPDATE`/`BEGIN` transaction in `put()` (retry at the operation boundary, not inside an open tx).
- Recommendation: **Build for v2.0 with FEAT-STORE-003 as one "Postgres connection hardening" change.** At minimum expose pool sizing + timeouts and attach the pool error handler (prevents process-crash on idle-connection errors). pg Pool already reconnects lazily, so prioritize the error handler + tunables over a custom retry loop.

### [FEAT-STORE-009] Expose dashboard / ops / inspector view-models on the public SDK
- Priority: MEDIUM
- Category: ux

A real bundled dashboard (FEAT-STORE-002) would consume these.

### [FEAT-STORE-010] S3 / object-store artifact adapter for large or high-volume artifacts
- Priority: LOW

- Scope: `src/contracts/index-store.ts` (the contract is already the seam — the gap is implementations + config to select them); a SQLite FTS5 index store (folds into FEAT-STORE-006) for full-text; optionally a vector index store; `docs/store-contracts.md:35-36` (align claim to reality). Effort: large  Breaking: no

- Recommendation: **Partial implement-or-retract.** v2.0: if SQLite (FEAT-STORE-006) ships, FTS5 makes the "full-text later" half true; **explicitly defer/soften the "vector" word** in `store-contracts.md:36` until there's a real plan.

## 4. Recommended v2.0 cut (opinionated — input to advisor review)

The thesis has two halves and a 2.0 that is a *security-hardening major* must do two things above all: (1) **make every documented promise true** (a hardening release with live claim-mismatches is self-defeating), and (2) **make the retrieval core actually rank** (it cannot today — the single widest promise/behavior gap). Everything else is sequenced behind those.

### MAKES v2.0

**Tier 0 — Contract honesty (the hardening release must not lie). All small, mostly additive.**
- **FEAT-CLI-001** (exit codes) — 3 lines, an active CI/cron hazard. Non-negotiable.
- **FEAT-AI-006** (one MCP error wire-shape) + **FEAT-AI-007** (spec annotation hints) + **FEAT-AI-003** (`next_valid_actions` everywhere) — the "make the AI surface contract-honest" cluster; two are claim-mismatches with exact file:line, all three small.
- **FEAT-CLI-002** (`--version`) + **FEAT-CLI-003** (TTY/NO_COLOR) — documented, trivial.
- **FEAT-RETR-003** (retract "vector," upgrade "full-text" once RETR-001 lands) — rides the RETR-001 PR; regen translations before tag.
- **FEAT-STORE-002** (retract the React-dashboard claim, delete the orphan `.d.ts`) — cheapest honest path.

**Tier 1 — The retrieval core (the thesis core).**
- **FEAT-RETR-001** (BM25) — the headline retrieval feature; pure-TS, no deps; makes "full-text" true.
- **FEAT-RETR-004** (scores + snippets in the bundle) — a ranker with nowhere to put the score is half a feature; ship together.
- **FEAT-RETR-006** (stop silently dropping ledger index hits — the *small* `MissingContext` safety fix) — serves the bundle's honesty contract.
- **FEAT-RETR-005**: **`offset`/pagination only**; defer `fields`/`phrase`.

**Tier 2 — Close the read-side lifecycle/provenance gaps the hardening opened.**
- **FEAT-VERSIONS-001** (gated version-history across kernel+SDK+MCP+CLI) — the seeded, load-bearing gap and the clearest legitimate flow forced into `/unsafe`. One kernel accessor unblocks all surfaces.
- **FEAT-SDK-009** (typed receipt/lineage reads — *verify-first*; build if not already present).
- **FEAT-AI-009** (`cluster_list_commands`) + **FEAT-SDK-002** (SDK pagination) — the agent's window into its own work queue + bounded reads.
- **FEAT-STORE-009** (expose view-models on the SDK, routed through the enforced stores) — small symmetry fix; also fixes the raw-store bypass.
- **FEAT-SDK-008** (re-audit package-boundary doc) — *discharged by* VERSIONS-001 + SDK-009; do it last in the slice.

**Tier 3 — Operator diagnostics (the operator half).**
- **FEAT-CLI-005** (`signals` command) — pull in; the hard part is already shipped, only the surface is missing. Makes the v1.x security spine observable.
- **FEAT-CLI-004 cheap tier** (rebuild/migrate `--progress` stderr counters) + the **FEAT-AI-004** `onProgress` plumbing on the SDK/ops layer (shared event shape) — kills the "is it hung?" pain.
- **FEAT-CLI-007** (`stats` counts) + **FEAT-CLI-008** (structured error envelope + exit-code map).

**Tier 4 — ONE storage headline (explicit product decision — pick exactly one).**
- **EITHER FEAT-STORE-006 (SQLite adapter)** — *my recommendation* — full four-store parity + atomic ledger + FTS5 in one server-less adapter; it simultaneously satisfies **FEAT-STORE-004** (ledger append), most of **FEAT-RETR-007** (scale), **FEAT-RETR-005** `fields`/`phrase`, and **FEAT-STORE-011** (full-text), and best serves the "embed a trustworthy cluster" framing.
- **OR FEAT-STORE-001 (complete Postgres parity)** + its dependents **FEAT-STORE-007** (migration runner) + **FEAT-STORE-003 / FEAT-STORE-008** (TLS + pool hardening, one pass) — if the user base is server-deployed.
- If Postgres is *not* the headline, still ship **FEAT-STORE-004** (NDJSON ledger append) standalone and **FEAT-STORE-003/008** if any Postgres users exist, because plaintext-credential risk and a torn ledger are thesis-level holes regardless of the headline.
- **FEAT-AI-005** (batch propose/resolve — *not* batch-commit) — captures the round-trip savings without diluting the write gate; include if time allows after the storage headline.

### DEFERS to v2.x
- **FEAT-RETR-002** (vector/embeddings) and therefore **FEAT-AI-008** (semantic MCP tool) — large, needs a pluggable local embedder; **hard sequence: RETR-001 → RETR-002 → AI-008.** Shipping the embedding field without the implementation is the worst of both worlds.
- **FEAT-AI-002** (cluster_diff) — only if VERSIONS-001 lands with room to spare; otherwise v2.x.
- **FEAT-SDK-003** (batch ingest/**commit**), **FEAT-SDK-006** (filtering), **FEAT-SDK-004 / FEAT-SDK-005** (*verify-first*; build in 2.0 only if the code is actually loose/heavy), **FEAT-SDK-007** (streaming reads).
- **FEAT-CLI-004 full tier**, **FEAT-CLI-009** (resumable rebuild), **FEAT-CLI-010** (`doctor --fix`), **FEAT-CLI-011** (`--yes` guard — fold into 2.0 only if CLI-003 lands).
- **FEAT-RETR-007** big adapter, **FEAT-STORE-005** (streaming artifact I/O — additive contract method *may* ride 2.0 if cheap), **FEAT-STORE-010** (S3 — blocked on per-store backend config), the *non-chosen* storage headline, **FEAT-STORE-011** vector half, **FEAT-CLI-007** latency histograms/OpenMetrics.

### Sequencing that must be respected
1. **FEAT-RETR-001 before FEAT-RETR-004** (no ranker → degenerate score) **and before the FEAT-RETR-003 wording change** (so "full-text" is true when claimed).
2. **FEAT-RETR-002 before FEAT-AI-008** (no vectors → nothing for the semantic tool to wrap). Both deferred — keep them paired.
3. **FEAT-VERSIONS-001 + FEAT-SDK-009 before FEAT-SDK-008** (the boundary-doc re-audit is discharged by them).
4. **One SDK accessor PR** (`listVersions/getVersion` + `drainSignals/peekSignals`) unblocks the MCP, SDK, and CLI facets of VERSIONS-001 and FEAT-CLI-005 at once — build it first in the lifecycle slice.
5. **FEAT-STORE-003 + FEAT-STORE-008 in one pass** (shared Pool/config surface); **FEAT-STORE-007 before/with FEAT-STORE-001** (parity adds tables the runner must manage).
6. **Decide the storage headline (SQLite vs Postgres-parity) before scheduling Tier 4** — it cascades into four other features. Do not build both.

**Rationale.** db-cluster's promise is *trust*. For an AI agent, trust = retrieval that ranks (RETR-001/004), a mutation loop whose errors are machine-legible and whose history is readable under the same gate (AI-003/006/007, VERSIONS-001, AI-009). For an operator, trust = diagnostics that fail loudly (CLI-001), a security spine you can *see* (CLI-005, CLI-007), and writes that don't silently ship plaintext or corrupt the ledger (STORE-003/004/008). And because 2.0 is the *hardening* major, a live claim-mismatch is a contradiction of the release's own thesis — so the Tier-0 honesty fixes are mandatory even though several are three-line changes. I deliberately keep vector retrieval and batch-commit OUT of 2.0: both are large, both have a correct-but-unbuilt substrate, and shipping either half-done would re-introduce exactly the "promised-not-delivered" gap this release exists to close. The single biggest open *product* question is the storage headline; my lean is SQLite-FTS5 because it makes four stores trustworthy at once with no server, but a server-deployed user base flips that to Postgres parity — that call belongs to the human.
