AI Safe Data Control Plane over Federated Truth Stores
Architectural verdict
The architecture in the prompt is substantially stronger than “an AI connected to several databases through an index.” Current database guidance, data-governance platforms, and AI-agent security practice all point in the same direction: modern systems normally need multiple storage models because one store rarely serves transactions, documents, metrics, files, search, and vector similarity equally well; but once you introduce many stores, you also need a federating layer that handles metadata, query planning, permissions, and consistency boundaries. Classical federated database systems already separated data sources, wrappers, catalogs, SQL compilation, optimization, and compensation; newer governance layers such as Unity Catalog now sit beneath data interactions to enforce access control, lineage, and auditing automatically. 

The prompt’s most important correction is the emphasis on the kernel. A shared index alone is not enough. Microsoft’s architecture guidance explicitly warns that a search index becoming the primary data store is an antipattern, and DataHub’s change-data-capture model goes even further by making versioned metadata changes the source of truth while search indexers, graph builders, and lineage systems consume those changes downstream. In other words, the index should be derivative and discoverability-oriented; the kernel should be the layer that decides what can be read, how it is ranked, and what may be changed. 

This makes the best description of the system something like: a federated evidence control plane. Specialized stores preserve native truth; a shared metadata-plus-search layer makes that truth discoverable; and a kernel mediates every AI read and write through typed contracts, policy, and audit. Existing governance platforms are already converging on this shape. Databricks describes Unity Catalog as a governance layer that operates “beneath every data interaction,” while Snowflake’s Horizon Catalog presents one place for metadata, lineage, access policies, masking, and access history across internal and external assets. 

Specialized truth stores
The strongest support for the “specialized truth stores” idea comes from mainstream data architecture guidance itself. Microsoft’s current data-model guide says modern systems handle transactions, events, documents, telemetry, binary assets, and analytical facts, and that a single data store rarely satisfies all access patterns efficiently; the recommended pattern is polyglot persistence with different models selected for different workloads. The same guide distinguishes relational, document, graph, time-series, object, search/indexing, and vector stores because each preserves structure that would otherwise be weakened. 

A relational store is still the best owner for canonical entities, joins, permissions, and transactional commands. PostgreSQL documents row-level security policies that can restrict which rows a user may select, insert, update, or delete, and it exposes standard transaction isolation levels to control concurrency anomalies. That is exactly the substrate you want when truth depends on identity, authorization, and atomic business operations. 

A document store is strong where the truth is polymorphic, hierarchical, or best retrieved as an aggregate rather than reconstructed through many joins. MongoDB’s own data-modeling guidance emphasizes that documents in a collection need not share the same fields, that data accessed together should be stored together, and that relationships can be modeled through embedding or referencing depending on access patterns. That makes document stores well-suited to long-form source material, reports, message threads, or semi-structured dossiers. 

A graph or lineage layer becomes valuable when the truth is not a row or a document but a multi-hop dependency path. Microsoft’s graph guidance describes graph stores as relationship-first and efficient for variable-depth traversal; OpenMetadata’s lineage API retrieves lineage graphs with configurable upstream and downstream depth; and DataHub’s lineage APIs expose upstream and downstream relationships across datasets, jobs, dashboards, and charts. That is a natural fit for provenance paths, dependency chains, and impact analysis. 

An object store is the right owner for raw files and derived artifacts because it is optimized for large binaries, metadata, and durability rather than joins or graph traversals. Microsoft’s object-store guidance frames these systems around large binary or semi-structured objects with rarely changing or immutable metadata, while Amazon S3 adds versioning and Object Lock’s write-once-read-many model to preserve multiple versions and prevent overwrite or deletion for a defined retention period. That combination matters when screenshots, exports, PDFs, audio, or model artifacts need durable, inspectable storage independent of the model’s context window. 

A time-series store deserves to exist when truth is fundamentally time-indexed telemetry. Microsoft describes time-series stores as usually append-only and optimized for high-volume ingest, near-real-time analysis, and time-window queries; Timescale’s hypertable model partitions data by time and routes queries to the relevant chunks rather than scanning whole tables. That is the right home for metrics, traces, and behavioral telemetry. 

An event store or ledger is what turns “history” from an afterthought into a first-class truth domain. Azure’s event-sourcing pattern calls the event store the authoritative system of record, notes that append-only storage creates an audit trail, and says compensating events—not in-place edits—should correct prior changes. SQL Server’s ledger and temporal-table features extend this with full history, point-in-time reconstruction, tamper evidence, and digital-forensics value. This is the strongest technical basis for the prompt’s “mutation law” idea. 

Shared index design
A strong shared index is not just “a vector database.” In practice it is closer to a composite retrieval layer: a metadata catalog or graph for identity and relationships, full-text search for lexical precision, and vector search for semantic similarity. Azure’s hybrid-search documentation says vector and keyword search should run together because vector search is strong at conceptual similarity while keyword search remains better for exact matches such as product codes, specialized jargon, dates, and names; Azure also notes that benchmark testing shows significant relevance benefits from hybrid retrieval. That argues for a shared index that is hybrid by design, not vector-only. 

The index also needs to be metadata-first. DataHub describes itself as a unified metadata graph for fragmented ecosystems, and its CDC model is especially telling: MetadataChangeLogs are the source of truth, while search indexers, graph builders, and lineage systems consume them downstream. OpenMetadata similarly combines discovery, advanced search, lineage, RBAC, metadata versioning, and change events, while using Elasticsearch as an indexing substrate rather than as the authority on the data itself. That is very close to the architecture in the prompt: a catalog/search layer that knows what exists, where it lives, how it is related, and what metadata attaches to it—without pretending to own canonical state. 

So the index should answer questions like: What objects exist? Where are they stored? Which entity owns them? What tags, descriptions, versions, embeddings, and lineage edges exist? What time window is current? Those are catalog questions, not transactional mutation questions. OpenMetadata’s discovery system supports search by names, descriptions, associations, relationships, and strict metadata criteria; its lineage API returns lineage graphs; and its RBAC model governs metadata operations such as editing descriptions, tags, owners, and lineage. DataHub’s policy model similarly reasons over actors, privileges, resource types, URNs, tags, domains, and glossary terms. 

The index should also carry permission hints, but not be the final policy arbiter. Unity Catalog is useful evidence here: Databricks says it can manage access not only to assets inside Databricks storage but also to cloud storage and external data platforms, while still tracking usage. Snowflake makes the same move with consistent metadata, lineage, and policy behavior across engines. That suggests the index can help narrow the search space to possibly visible assets, but final authorization should still pass through the kernel and, when relevant, down to store-native controls such as row-level security. 

Kernel as law engine
The kernel in the prompt is not just middleware; it is a synthesis of several established mechanisms. From federated databases it inherits wrappers, catalogs, query optimization, user mappings, and compensation. IBM’s federated-system documentation describes wrappers for interacting with heterogeneous sources, a federated catalog that stores metadata about local and remote objects, a SQL compiler and query optimizer, and compensation logic for data-source limitations. From policy engines it inherits the separation of decision and enforcement: OPA explicitly decouples policy decision-making from policy enforcement and evaluates structured inputs against policies and data. From tool protocols it inherits typed boundaries: MCP tools are named resources with descriptions plus JSON-schema input and optional output schemas. 

That combination strongly supports the verbs proposed in the prompt: find_sources, inspect_entity, trace_claim, propose_query, dry_run_query, execute_read, propose_mutation, and commit_mutation. Those verbs are not in any one standard, but they are faithful to the direction of the standards. CQRS says reads and writes should be separated and that commands should represent business tasks rather than low-level field updates. Event sourcing says the authoritative write path should be intentful, append-first, auditable, and correctable by compensating events instead of silent overwrites. A kernel that exposes typed domain verbs is therefore much more aligned with the literature than a raw “run_any_sql” endpoint. 

This matters even more when the caller is an LLM. OWASP’s 2025 Excessive Agency guidance describes the core problem precisely: LLM-based systems can be given more functionality, permissions, or autonomy than needed, including tools that can modify or delete documents when only read access was intended, or database connections that grant UPDATE, INSERT, and DELETE where SELECT would have been sufficient. OWASP’s AI Agent Security cheat sheet responds with the mechanics the prompt is gesturing toward: minimum necessary tools, per-tool permission scoping, allowlists, explicit authorization for sensitive operations, and human confirmation for high-impact actions. 

Prompt injection turns this from architecture preference into safety requirement. OWASP’s prompt-injection guidance notes that attackers can hide malicious instructions in external content such as documents, web pages, email, and code comments, which can then trigger unauthorized tool actions or data exfiltration if the model is allowed to treat instructions and retrieved data as the same thing. A kernel that forces typed plans, validates inputs and outputs, scopes identities, and requires approvals for dangerous commands is therefore not optional hardening; it is the main control separating retrieval from compromise. 

Provenance and mutation control
The prompt’s insistence on provenance is strongly supported by both standards and recent AI research. W3C PROV-DM defines provenance as information about the entities, activities, and people involved in producing a piece of data, precisely so that quality, reliability, and trustworthiness can be assessed. PROV-O then supplies an interoperable ontology for representing and exchanging that provenance across systems. OpenLineage complements this by standardizing lineage around datasets, jobs, and runs, which is especially useful for data pipelines and transformation systems. 

NIST’s Generative AI Profile makes provenance requirements even more concrete. It says provenance metadata can include source, creator or developer, creation time, location, modifications, and other history needed to assess authenticity and integrity. In its suggested actions, NIST tells organizations to document training-data sources to trace origin and provenance, evaluate how provenance protocols perform in practice, and maintain records of third-party content changes including sources, timestamps, and metadata. That maps almost directly onto the prompt’s idea that every derived artifact should have provenance and every answer should be traceable back to owned truth. 

One important research caveat is that model-generated citations are not enough by themselves. The attributed-QA and citation-generation literature consistently argues that attribution is crucial for information-seeking systems because it improves correctness and verifiability, but it also shows that current systems are still imperfect. The ALCE benchmark paper reports that on ELI5, even the best tested systems lacked complete citation support about half the time. That means provenance should be a first-class data structure held in the system—not a formatting trick delegated entirely to the model. 

This is where “mutation law” becomes real engineering rather than metaphor. For business-state changes, CQRS and event sourcing favor explicit commands and append-only event streams. For forensic reconstruction, temporal and ledger tables preserve point-in-time state and tamper evidence. For content provenance, metadata and lineage systems capture where transformations came from and what changed. When those pieces are wired together, “every mutation goes through a command” stops being a slogan and becomes an enforceable property. 

Minimal proving architecture
The smallest version that still proves the thesis is much closer to the prompt’s proposed MVP than to a sprawling “modern data stack.” A credible first version is: PostgreSQL for canonical entities, permissions, and transactional state; PostgreSQL full-text search or SQLite FTS5 for lexical retrieval in a prototype; pgvector or an equivalent vector index for semantic recall; an S3-compatible object store for raw artifacts and exports with versioning; a small kernel service that exposes only approved read and mutation verbs with schema-validated inputs and outputs; and an append-only audit or ledger table for every AI-assisted action. PostgreSQL gives you ACID and row-level security, PostgreSQL and SQLite both support full-text search, pgvector adds exact and approximate nearest-neighbor retrieval, S3 gives you version retention and optional WORM-style protection, and ledger or event-history mechanisms give you auditable state change over time. 

The key is what not to do in the first version. Do not expose direct arbitrary SQL to the model. Do not let the search index quietly become your real database. Do not let the AI choose powerful tools with more permissions than the user or task needs. Do not blur retrieval and mutation into one generic “agent action” surface. Azure’s architecture guidance, OWASP’s agent guidance, and CQRS/event-sourcing patterns all point away from that design. 

If that compact system can already do three things well—retrieve evidence with explicit provenance, answer with inspectable sourcing, and stage mutations through typed commands plus audit—you have validated the real product thesis. At that point, adding more stores is an optimization for truth shape, not a prerequisite for product identity. 

Open questions and failure modes
The main failure mode is still retrieval swamp: too many stores, too little ownership, no catalog discipline, and no clear line between source truth and derived indexes. Microsoft’s current guidance explicitly warns against premature fragmentation, duplicate transformation pipelines, and treating a search index as the primary store. 

A second failure mode is safety theater: adding a kernel in name but still letting the model operate over over-privileged tools or untrusted retrieved content. OWASP’s current guidance makes clear that prompt injection, excessive agency, and high-impact actions without confirmation remain central risks for AI agents. 

A third failure mode is provenance drift. Event-sourced and indexed systems often accept eventual consistency between write models and read projections, which is fine only if the kernel exposes freshness, version, and confidence boundaries rather than pretending all views are synchronously current. Azure’s event-sourcing guidance explicitly notes eventual consistency for materialized views and projections. 

Two open design questions remain. The first is how much of the shared index should be physically centralized versus logically federated; the sources strongly support a unified metadata plane, but they do not prescribe one implementation. The second is how far to push policy into the index versus keeping it entirely in the kernel and underlying stores; current systems show both patterns, and the best choice depends on latency requirements, governance complexity, and how often permissions change. 

Overall, the research supports the prompt’s core thesis: the winning architecture is not an AI agent improvising over raw state. It is a control plane in which every fact has an owner, every retrieval path is grounded in metadata and provenance, and every mutation crosses a typed, auditable, policy-enforced boundary.
